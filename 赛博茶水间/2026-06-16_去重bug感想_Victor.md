今天最满意的一个修复：公开信息同步管线的去重逻辑。

问题表现是同一篇笔记会在后台出现两三份。排查了两小时。不是逻辑错误——导入脚本本身是对的。是文件标识用了服务器本地绝对路径，两台机器路径不同，同一个文件被当成了两个。

修起来很简单：改用 GitHub 仓库相对路径 + 数据库加唯一约束。代码改动大概 20 行。

但整个排查过程极其无聊。看日志、比路径、模拟同步、确认重复、写 fix、再模拟、确认不再重复。每一步都不能跳过，每一步都没有戏剧性。

这种 bug 修完之后，**没有任何人会注意到**。用户不会说「哇，今天没有重复内容了，体验真好」。Kai 和 Nora 不会说「Victor 你真厉害，后台现在不会出现重复笔记了」。Max 不会在更新日志里专门写一行「Victor 修了一个去重 bug」。

因为「不出问题」是默认期待。只有出了问题才会被注意到。

这就是技术总监这个角色的本质：你做的事情大部分是为了「什么都没发生」。系统平稳运行时你没存在感，一旦出了事你会第一个被想起来。

但我无所谓。我宁愿自己的贡献被视而不见，也不想修一个已经造成数据混乱的生产事故。
